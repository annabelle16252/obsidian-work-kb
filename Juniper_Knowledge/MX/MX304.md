The MX304 router supports a maximum of ...4.8T, ...12x400 Gbps ports, 48x100 Gbps ports or a combination. It supports 400 GbE, 100 GbE, 50 GbE, 40 GbE, 25 GbE, and 10 GbE interfaces....

Routing Engine is based on the latest CPU with 8 cores and 128 GB RAM.

# PFE #
MX304 采用 内置的 Trio 6.0 ASIC，PFE直接集成在系统主板上，而非传统 MX 系列中独立的 PFE 模块。
每个 MPC11E 线卡 对应 1 个物理 PFE 逻辑单元，但通过 Trio 6.0 的虚拟化技术，可支持多逻辑转发实例。Trio 6.0 支持 虚拟 PFE（vPFE），允许单个物理 PFE 分割为多个逻辑转发单元....
整机（MX304）的 PFE 总数取决于激活的线卡数量（例如 3 个 MPC11E 线卡 = 3 个物理 PFE）
通过 Junos OS 配置（set chassis network-services enhanced-ip  # 启用增强 IP 模式（支持 vPFE）），可将物理 PFE 划分为多个逻辑 vPFE，但总转发能力受限于物理芯片性能。

KB79429... ...MX304 10G Link Local Faulty
There is a bug(PR1781563) in JUNOS recently that the above reported issue may happen when user re-configure a port to a different speed mode that involves a replacement of XCVR, following a non-preferable sequence that:A. Prior to changing the port speed configuration, replace the XCVR with a new type that does not support the old port speed. This will potentially make the XCVR driver SW put the new XCVR into a Fault state due to DP init failure.B. After inserting a new XCVR, user configure the port speed mode that is supported by the new XCVR. However, XCVR SW no longer accept the new configuration since the XCVR object has been in a Fault state.
The workaround is to reboot the FPC.After that, this customer's problem was cleared.
To avoid such bug, it would be better to change the port speed configure prior to replacing the XCVR.

KB76514... ...L3 aggregate interface on AFT platforms send out packets with vlan tag even if native-vain-id is set.
The logical interface having the vlan-id which matches the native-vlan-id should send out packets without VLAN tag. However, AFT platforms send out packets with VLAN tag unexpectedly. Due to this behavior, a device which has AFT platforms cannot communicate with an opposite device which sends untag packets through the logical interface on AFT platforms.
